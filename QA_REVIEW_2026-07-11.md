## 3. Safety-critical invariants — verified

### 3a. Red-never-VO2 (Decision #61 / #30, scoped by #161)
The delivery guarantee holds. `blocks_red_vo2(verdict, ir)` is a pure predicate
enforced hard at all three **coach-adjustment** delivery boundaries:

- auto-push (`executable_coaching.py:408`) → blocks + audits, `continue`
- `send_today` (`:518`) → 409
- `approve_adjustment` (`:735`) → 409

Batch 88 deliberately makes Mark's **explicit manual authoring** path soft:
`add_workout`/`edit_structured_workout` call `build_freeform_bike_workout(..., soft_gates=True)`
and `_append_red_vo2_warning(...)`, then deliver the as-planned baseline via
`reconcile_deliveries → _deliver_one` — which by design (#99) carries no Red-VO2
block. **This is internally consistent**, not a hole: the `red_vo2` warning
truthfully says "sending it because you built it," and the baseline path was
never gated for the manual case even before Batch 88. The coach/automated path
keeps `soft_gates=False` (default) → hard 422. ✔️

### 3b. Subjective is downgrade-only (a cheery check-in can't lift the day)
`_morning_verdict` (`morning_analysis.py:1255`) checks the Red conditions
(age-adj sleep < 60; HRV below baseline + low/unbalanced) **first**, so subjective
never bypasses a Red. Subjective only appears as a *downgrade* (`< 5 → Amber`) or
as a **gate** inside `_soft_sleep_recovery_override` (`:1452`), where
`subjective_score >= 5` is required but never used to boost. The override itself
only fires in the 60–74 soft-sleep band, never on a Red day. ✔️

## 6. Findings (ranked)

### Finding 1 — RLS disabled on 18 health-data tables in a shared Supabase project · **Security · High (known/tracked)**
Supabase security advisor returns a **critical** `rls_disabled` lint. 18 of 23
`coach.*` tables have Row-Level Security **off**, including all of Mark's health
data: `sleep`, `daily_metrics`, `analyses`, `activities`, `activity_timeseries`
(545k rows), `manual_entries`, `planned_workouts`, `plan_blocks`,
`temperature_readings`, `metric_baselines`, `weather_daily`, etc. Only the
auth/notification tables (`profiles`, `refresh_tokens`, `push_subscriptions`,
`notification_preferences`, `audit_log`) have RLS on.

This matters more here than usual because **the same Supabase project also hosts
a separate public-facing "movie" app** in the `public` schema (whose frontend
ships a public anon key). If the `coach` schema is reachable through the Supabase
Data API, that public anon key could read/modify Mark's health data.

**Verified during follow-up (severity is latent, not open):** garmin-coach's own
app does **not** use the Supabase anon key — its React frontend talks only to the
FastAPI backend (JWT auth, direct asyncpg connection that bypasses RLS). A direct
privilege check confirms the `anon` and `authenticated` roles hold **no grants**
on the `coach` schema (`has_schema_privilege('anon','coach','USAGE') = false`, and
`false` for SELECT/INSERT on the tables); the REST Data API is additionally
egress-restricted (HTTP 402) right now. So **nothing is actually reachable via the
Data API today** — the real control holding the line is the absence of role
grants. This is a **defense-in-depth gap and a landmine**, not an open breach: a
future stray `GRANT` or an "Exposed schemas" change would leak the data with RLS
off.

**Fix in progress — this is now Batch 89** (`fix/batch-89-coach-rls`, Decision
#162). Migration `015_coach_rls.py` enables RLS on the 18 tables with **no
policies** (the backend connects as the `postgres` owner and bypasses RLS —
proven by `profiles` reads working in prod despite 001's `auth.uid()` policies —
so RLS-with-no-policy is deny-all for `anon`/`authenticated` and a no-op for the
app), wrapped in migration 001's `IF EXISTS (auth schema)` guard so CI stays
green. Post-deploy step: recheck the Supabase advisor clears.

This was STATUS open item (b); this review sharpened the exposure rationale and
scoped the fix. Remediation ref:
<https://supabase.com/docs/guides/database/postgres/row-level-security>

### Finding 3 — Soft/abs band constants duplicated Python ↔ TypeScript · **Maintainability · Low**
`45 / 150 / 1 / 300 / 480` live independently in
`structured_workout_builder.py` and `StructuredWorkoutSheet.tsx`. They match
today, but a future change to one side would silently desync the live preview
from the server response. Consider lifting them into `packages/shared`.

### Finding 4 — `HTTP_422_UNPROCESSABLE_ENTITY` deprecation · **Tech-debt · Low**
`workout_delivery.py` (and the pytest run, 11 warnings) uses the Starlette
constant deprecated in favour of `HTTP_422_UNPROCESSABLE_CONTENT`. Cosmetic now;
a breaking removal on a future Starlette major. One-line swap when touched.

### Finding 6 — `pip-audit` can't run in this sandbox · **Environment**
