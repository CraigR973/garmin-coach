### 2026-06-20 ‚Äî Phase 0b (hosting)
34. **Single Supabase project for both movie app and garmin-coach, isolated by schema.** Movie app uses `public`; garmin-coach uses `coach`. Free tier allows only 2 active projects; this avoids using a second slot. *Why:* trivial for a 1-2 user private app where schemas don't conflict; Alembic targets `coach` via `version_table_schema="coach"` and `SET search_path TO coach, public`.
35. **Supabase session-mode pooler (port 5432), not transaction mode (port 6543).** Railway containers are IPv6-only; Supabase's direct host (`db.*.supabase.co:5432`) is IPv4-only ‚Üí "Network is unreachable". Supabase Supavisor pooler (`aws-1-eu-north-1.pooler.supabase.com`) is dual-stack. Session mode maps 1:1 to a PG backend for the connection lifetime ‚Äî prepared statements persist correctly.
36. **Vercel `vercel.json` at repo root (not `apps/web/`).** Vercel's monorepo auto-detection picks `vercel.json` from root; placing it in `apps/web/` caused `@coach/shared: workspace:*` to fail (Vercel used npm, not pnpm). Root config sets `buildCommand: "pnpm --dir apps/web build"` + `installCommand: "pnpm install --frozen-lockfile"`. *Why:* pnpm workspaces require pnpm; only root-level `vercel.json` lets us override the install command.
37. **Railway NOT connected to GitHub auto-deploy (deliberate for now).** Deploying via `railway up` (builds from local source and uploads). Connect to GitHub in Railway dashboard Settings > Source Repo when continuous deployment is wanted. *Why deferred:* faster to ship; auto-deploy can be wired once Phase 1 is stable.

### 2026-06-24 ‚Äî Scheduler reliability (Hive feed stall)

86. **The in-process APScheduler is treated as best-effort, not the source of truth for scheduled work: jobs are made runnable standalone (`python -m src.run_scheduled <job>`) so an external scheduler can drive them, and interval jobs are seeded to fire shortly after startup.** Root cause of the stalled Hive feed (only 2 readings ever, both manual): every scheduled job runs in-process inside the web container (`main.py` lifespan ‚Üí `create_scheduler`), which is only reliable if the container runs continuously ‚Äî and in prod it does not (frequent restarts / idle-cycling), so the plain 15-minute Hive interval rarely reached its first fire (start + 15 min) while the +5-minute-seeded activity poll occasionally did, and the wall-clock jobs (06:30 morning sync, evening nudges) only fired if the container happened to be awake. The poll itself is healthy: a manual run writes a reading immediately, and the token / profile / Hive API all work. Fix is two-part: (1) **band-aid** ‚Äî seed the Hive poll with `next_run_time = now + 2 min` (mirroring the activity poll) so a freshly started container polls within ~2 min instead of waiting a full interval (the test asserts both interval jobs are seeded); (2) **durable** ‚Äî a new `src/run_scheduled.py` runs any single named job to completion and exits, mapping job names to the existing `run_*` coroutines (identical behaviour, no duplicated logic), so jobs can be driven by Railway Cron / GitHub Actions / a manual `railway run` independent of the web container's lifecycle; `docs/runbooks/scheduled-jobs-cron.md` documents the schedules and the cutover (`SCHEDULER_ENABLED=false` once external cron is verified, since the jobs are idempotent and can overlap briefly). *Why:* binding all scheduling to a single web container's uptime is the actual defect; decoupling execution from that container ‚Äî via a standalone runner that reuses the scheduler functions ‚Äî makes scheduled work survive restarts/idle-cycling, and the band-aid ships immediate relief without waiting for the cron wiring. **DST caveat:** Railway cron is UTC and does not track Europe/London BST‚ÜîGMT, so the timezone-agnostic interval jobs (`hive-poll`/`activity-poll`/`backup`) move to cron cleanly, but the wall-clock jobs are best kept on in-process APScheduler (which handles DST via `timezone=Europe/London`) with the container made always-on ‚Äî or accept ¬±1h drift. **Open:** whether to fully retire in-process APScheduler or run a hybrid is deferred until external cron is wired and the always-on / Railway "App Sleeping" question is settled in the dashboard.

**Correction (2026-06-24, same day):** after Railway **App Sleeping was disabled** (the container now stays up ‚Äî confirmed: the +5-min activity poll fired at 12:52 and the 15-min Hive interval fired *on schedule* at 13:02), the Hive poll's *real* prod failure finally surfaced in the logs: `ModuleNotFoundError: No module named 'pyhiveapi'`. **`pyhiveapi` was missing from `apps/api/requirements.txt`** (the Dockerfile installs only that file), so the poll fired in prod but raised immediately ‚Äî and every earlier "a manual run writes a reading" check passed *only* because `railway run` executes against the **local venv**, which had `pyhiveapi` installed ad-hoc and never tracked. Disabling App Sleeping was still necessary (it makes the container stay up so the scheduler fires at all), and the seed / `run_scheduled` changes remain valid resilience. **Lesson:** `railway run` validates code against the *local* venv, not the deployed image ‚Äî verify dependency-sensitive prod behaviour from the container logs, not a local invocation.
