### 2026-06-21 — Phase 2 Batch 14 (dynamic weekly restructuring)
63. **The weekly restructurer is a deterministic permutation engine over the week's bike sessions, with the no-stack rule as a hard constraint and defer-on-fatigue as a lexicographic objective.** `plan_week_restructure` (in `services/weekly_restructure.py`) reassigns the week's bike workouts among the dates that already hold a bike session (strength/mobility days stay put and act as spacers), enumerates the permutations (capped at `MAX_PERMUTE_SESSIONS`), and picks the lowest-cost feasible arrangement. **Feasibility** = no VO2↔Sweet-Spot pair within `MIN_GAP_DAYS` (same/adjacent days). When **fatigued**, the cost key is `(infeasible?, defer_cost, moves, shift)` so deferring hard sessions later in the week is the primary objective; when fresh it is `(infeasible?, moves, shift, defer_cost)` so a conflict is fixed with minimal disruption and an already-spaced fresh week is a no-op. *Why:* the spacing/fatigue rules are documented coaching constraints (KB `training_plan.constraints`, ARCHITECTURE §6) and a small enumerated search makes them inspectable, deterministic, and unit-testable without a DB or an LLM — the engine is a pure function and the DB service is a thin wrapper. The fatigue signal itself (`assess_recovery_signal`) reads the latest readiness, HRV status, and the morning-verdict trend (a Red, ≥2 Ambers in the window, low readiness, or unbalanced HRV).
64. **Restructures are human-triggered and approval-gated, not auto-applied by the scheduler; the plan is versioned on `planned_workouts` and reuses the Batch 12/13 delivery rail (no new migration).** `apply_for_week` versions only the changed dates (new active version per date, prior deactivated — same pattern as `coaching_state.override_planned_workout`), audits the restructure as an `analyses` row (`analysis_type="weekly_restructure"`, `subject_date=week_start`), and proposes the changed bike workouts via `WorkoutDeliveryService.propose_from_ir` with `origin="weekly_restructure"` — they stay `proposed` until the human approves (Decision #29). It is exposed through `GET/POST /api/v1/restructure/*`, not wired into a scheduler job. *Why:* a scheduler that silently rewrote the plan and queued deliveries would violate the "never silent, mould-it-then-lock" contract (#16/#29); making restructure an explicit human action (preview → apply → approve → push) keeps Mark in control. Versioning per changed date preserves plan history and satisfies the `(user, date, version)` constraint, and reusing the existing proposal/audit homes (#54, #62) keeps Batch 14 migration-free.
65. **The VO2 progression — including Rønnestad 30/15 — is centralized in a shared `services/vo2_progression.py` toolkit used by both the plan seed and the restructurer.** `select_vo2_protocol(week_number, block_type)` returns 30/30 for early build and Rønnestad 30/15 from `RONNESTAD_FROM_WEEK` (7) onward; `build_vo2_structured_workout` emits the `structured_workout` dict carrying the documented constraints (`ergMode="off"`, even-paced 105-110% FTP / 15s easy — Decision #33). `coaching_state._build_templates` now calls the toolkit instead of inlining the VO2 shapes, and when `apply_for_week` defers/moves a late-build VO2 session it regenerates it through the toolkit so the deferred session keeps the 30/15 / ERG-off constraints. *Why:* the seed and the dynamic restructurer must agree on which protocol a given build week uses; a single source removes drift and makes the 30/15 progression independently testable, and it is the natural home for Batch 16's generator to draw VO2 days from later.

### 2026-06-21 — Phase 2 Batch 15 (holiday pause/resume)
66. **A holiday window is treated as a recovery-week equivalent: planned workouts inside the window are versioned as `status='skipped'` with `source='holiday_pause'`, not deleted.** `HolidayPauseService.pause` deactivates the existing `planned_workouts` rows and inserts new versions at `version+1` carrying the skipped status (same versioning pattern as Batches 13/14). *Why:* versioning preserves plan history and audit traceability; the `(user_id, workout_date, version)` uniqueness constraint stays intact; and it mirrors how the weekly restructurer and workout-delivery service handle plan changes — a consistent write pattern across all plan-mutation services.
67. **On return from holiday the 2121 block continuation rule is applied: Build1 pre-holiday → first week back continues as Build2; Build2 pre-holiday → first week back repeats Build1.** `continuation_label`/`continuation_week_number` are pure deterministic helpers — Build1/Build2 is determined by `(sequence_index - 1) % 3 == 0` (Build1) / `== 1` (Build2). `resume` finds the nearest build block that ended before the holiday, resolves the label and week-number, then regenerates the first post-holiday build block's workouts from `_block_templates` via `_regenerate_block`. *Why:* the 13-week 2121 slate is the core training block structure; the holiday pause must not arbitrarily reset to week-1 content when a partial build pair (Build1 or Build2) was in progress before the break — doing so would either repeat identical sessions or skip a progression step. The mirror pattern (came from Build1 → continue to Build2; came from Build2 → redo Build1) gives Mark continuity with minimal lost work and is the coaching-logic equivalent of how the existing slate handles recovery weeks.
68. **Holiday windows are stored as JSONB in the existing `knowledge_base` table at `section='holiday_windows'`; no migration is required.** Content schema: `{"windows": [{startDate, endDate, pausedAtUtc, resumedAtUtc}]}`. A new KB row is versioned (existing deactivated, new active) on each pause and each resume, preserving window history. *Why:* mirrors the Batch 13/14 pattern — using `knowledge_base` for structured operational context avoids table proliferation and any Alembic migration; the JSONB column and versioning semantics already suit append-append history. An active window (no `resumedAtUtc`) is the guard that prevents double-pause.

### 2026-06-22 — Phase 2 Batch 16 (app-generated 13-week blocks)
69. **The 13-week block generator is deterministic, not an LLM call, and reuses the shared `coaching_state` block templates + the Batch 14 `vo2_progression` toolkit.** `generate_block_plan` (in `services/block_generator.py`) maps the fixed `BLOCK_SEQUENCE` (2 build / 1 recovery ×3, wk12 taper, wk13 consolidation) over concrete dates from a start date + FTP, drawing each week's workouts from `_block_templates`; generated VO2 days therefore carry 30/30 early and Rønnestad 30/15 from ~Wk7 automatically. *Why:* the acceptance criteria are all *structural, testable invariants* — the exact 2121 shape, the VO2 progression rule, and refine/lock versioning — and the rest of v2 (Batches 13/14) already chose deterministic engines over LLM calls for plan-shaping safety properties. A deterministic generator makes the shape inspectable and unit-testable, guarantees it without depending on `ANTHROPIC_API_KEY`, and keeps a single source of truth for VO2 protocol selection shared with the seed and the restructurer (#65). "Conversational/refine" (Decision #16) is delivered as the *iterative refine-then-lock loop* the human drives, not as free-text generation.
70. **The generated block is a refine-then-lock draft stored as JSONB in `knowledge_base` at `section='generated_block'`; only `lock` mutates `planned_workouts`, and no migration is required.** Lifecycle: `generate` (status `draft`) → `refine*` (edit individual days) → `lock` (status `locked`, writes the plan) — each step versions the KB row (max-version + 1, so a discard-then-regenerate never reuses a version number). `generate` returns 409 if an unlocked draft already exists, and `discard` drops an unlocked draft, so refinements are never silently clobbered (Decision #29, "mould it, fix errors"). `lock` writes one versioned `plan_block` per week (block name versioned to avoid colliding with the seed slate) and versions up any active `planned_workout` on each date — the same write pattern as the holiday regenerator — so locking integrates with, rather than duplicates, plan history. Once locked the rows are ordinary active `planned_workouts` with `source='block_generator_lock'`, so they feed the daily loop and the Zwift rail under the existing approve → push gate (16.4) with no extra wiring. Exposed via `GET/POST /api/v1/block-generator/*` (human-driven) and a `/builder` PWA tab. *Why:* mirrors the Batch 15 no-migration JSONB pattern; keeping the draft out of `planned_workouts` until lock means a half-moulded block never leaks into the daily verdict or a delivery proposal, and reusing the existing versioning/delivery homes (#54, #64) keeps Batch 16 migration-free.
