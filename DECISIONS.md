### 2026-06-21 — Phase 2 Batch 14 (dynamic weekly restructuring)
63. **The weekly restructurer is a deterministic permutation engine over the week's bike sessions, with the no-stack rule as a hard constraint and defer-on-fatigue as a lexicographic objective.** `plan_week_restructure` (in `services/weekly_restructure.py`) reassigns the week's bike workouts among the dates that already hold a bike session (strength/mobility days stay put and act as spacers), enumerates the permutations (capped at `MAX_PERMUTE_SESSIONS`), and picks the lowest-cost feasible arrangement. **Feasibility** = no VO2↔Sweet-Spot pair within `MIN_GAP_DAYS` (same/adjacent days). When **fatigued**, the cost key is `(infeasible?, defer_cost, moves, shift)` so deferring hard sessions later in the week is the primary objective; when fresh it is `(infeasible?, moves, shift, defer_cost)` so a conflict is fixed with minimal disruption and an already-spaced fresh week is a no-op. *Why:* the spacing/fatigue rules are documented coaching constraints (KB `training_plan.constraints`, ARCHITECTURE §6) and a small enumerated search makes them inspectable, deterministic, and unit-testable without a DB or an LLM — the engine is a pure function and the DB service is a thin wrapper. The fatigue signal itself (`assess_recovery_signal`) reads the latest readiness, HRV status, and the morning-verdict trend (a Red, ≥2 Ambers in the window, low readiness, or unbalanced HRV).

### 2026-06-23 — Phase 3 Batch 22 (hypothesis evaluation)

83. **Hypothesis evaluation is a *deterministic, advisory* engine that reuses the Batch 17 insights math (`_slope`/`pearson`/`compute_drivers`) to recommend a conclusion per experiment, but never changes status — concluding stays the human-gated, terminal `POST /…/status` action (#72). ### 2026-06-24 — Scheduler reliability (Hive feed stall)

86. **The in-process APScheduler is treated as best-effort, not the source of truth for scheduled work: jobs are made runnable standalone (`python -m src.run_scheduled <job>`) so an external scheduler can drive them, and interval jobs are seeded to fire shortly after startup.** Root cause of the stalled Hive feed (only 2 readings ever, both manual): every scheduled job runs in-process inside the web container (`main.py` lifespan → `create_scheduler`), which is only reliable if the container runs continuously — and in prod it does not (frequent restarts / idle-cycling), so the plain 15-minute Hive interval rarely reached its first fire (start + 15 min) while the +5-minute-seeded activity poll occasionally did, and the wall-clock jobs (06:30 morning sync, evening nudges) only fired if the container happened to be awake. The poll itself is healthy: a manual run writes a reading immediately, and the token / profile / Hive API all work. Fix is two-part: (1) **band-aid** — seed the Hive poll with `next_run_time = now + 2 min` (mirroring the activity poll) so a freshly started container polls within ~2 min instead of waiting a full interval (the test asserts both interval jobs are seeded); (2) **durable** — a new `src/run_scheduled.py` runs any single named job to completion and exits, mapping job names to the existing `run_*` coroutines (identical behaviour, no duplicated logic), so jobs can be driven by Railway Cron / GitHub Actions / a manual `railway run` independent of the web container's lifecycle; `docs/runbooks/scheduled-jobs-cron.md` documents the schedules and the cutover (`SCHEDULER_ENABLED=false` once external cron is verified, since the jobs are idempotent and can overlap briefly). *Why:* binding all scheduling to a single web container's uptime is the actual defect; decoupling execution from that container — via a standalone runner that reuses the scheduler functions — makes scheduled work survive restarts/idle-cycling, and the band-aid ships immediate relief without waiting for the cron wiring. **DST caveat:** Railway cron is UTC and does not track Europe/London BST↔GMT, so the timezone-agnostic interval jobs (`hive-poll`/`activity-poll`/`backup`) move to cron cleanly, but the wall-clock jobs are best kept on in-process APScheduler (which handles DST via `timezone=Europe/London`) with the container made always-on — or accept ±1h drift. **Open:** whether to fully retire in-process APScheduler or run a hybrid is deferred until external cron is wired and the always-on / Railway "App Sleeping" question is settled in the dashboard.

**Correction (2026-06-24, same day):** after Railway **App Sleeping was disabled** (the container now stays up — confirmed: the +5-min activity poll fired at 12:52 and the 15-min Hive interval fired *on schedule* at 13:02), the Hive poll's *real* prod failure finally surfaced in the logs: `ModuleNotFoundError: No module named 'pyhiveapi'`. **`pyhiveapi` was missing from `apps/api/requirements.txt`** (the Dockerfile installs only that file), so the poll fired in prod but raised immediately — and every earlier "a manual run writes a reading" check passed *only* because `railway run` executes against the **local venv**, which had `pyhiveapi` installed ad-hoc and never tracked. So the primary Hive-specific blocker was the **missing dependency**, not the scheduler; fixed by adding `pyhiveapi>=0.5.16` to `requirements.txt`. Disabling App Sleeping was still necessary (it makes the container stay up so the scheduler fires at all), and the seed / `run_scheduled` changes remain valid resilience. **Lesson:** `railway run` validates code against the *local* venv, not the deployed image — verify dependency-sensitive prod behaviour from the container logs, not a local invocation.

### 2026-06-24 — Wake-triggered morning verdict

87. **The morning run fires when Mark *actually wakes* (detected from his Garmin `sleepEnd`), not on a fixed 06:30 cron — in-process via a `wake_check` interval job that polls sleep every ~15 min and triggers the *unchanged* `run_morning_weather_sync` once his wake is stable, with a 09:30 backstop.** The backfill made the case quantitative: across 363 nights his median wake is **08:22** (range 03:45–09:24) and he wakes after 06:30 on **98.6 %** of mornings, so the legacy 06:30 cron fired ~2 h before he woke almost every day — reading a pre-dawn placeholder for time-of-day-live Training Readiness and an unfinalized sleep score. **Mechanism:** a new pure decision core `services/wake_detection.py::is_morning_ready(today, sleep, *, prev_sleep_end, now, backstop, duration_floor_min, settle_min) → WakeDecision{fire|wait|nap_ignored, sleep_end_to_persist, reason}` (no I/O, exhaustively unit-tested), driven by `scheduler.run_wake_check()`. Per poll, per active profile, within the window (03:30–10:00 Europe/London): short-circuit if today's morning analysis already exists (cheap `analyses` read, no Garmin call — stops polling for the rest of the day); else a **light sleep-only Garmin poll** (new `GarminConnectClient.fetch_sleep` = the single `get_sleep_data` call, not the ~10-call `fetch_daily_payloads`); apply the **back-to-sleep stability guard**; persist the current `sleepEnd` as a `wake_check` audit row for the next poll's comparison; if ready, run `run_morning_weather_sync` once. **The stability guard is the key correctness point:** Garmin records one consolidated overnight session (brief awakenings are awake epochs *inside* it), but a genuine two-block night can close the first session early — so we never fire on first detection. Fire only once today's `sleepEnd` (a) equals the value persisted on the previous poll (unchanged ≈ stable across two consecutive ~15-min polls) **and** (b) sits ≥ `settle_min` (20 min) in the past **and** (c) the session clears a `duration_floor_min` (180 min) that excludes naps. A *later* `sleepEnd` than last poll means he drifted back to sleep → keep waiting and persist the later value until it settles at his true get-up. Cost: the verdict lands ~15–20 min after wake rather than instantly — a fair trade for never building it on a half-night. **Backstop (belt-and-suspenders):** a dedicated `morning_backstop` cron at 09:30 Europe/London runs `run_morning_weather_sync` directly, **and** `is_morning_ready` fires on `now ≥ backstop` regardless of the data, so a verdict is guaranteed even if wake was never detected (watch not worn / never synced / container down through the window) whether the in-process scheduler or external cron drives it. **State is migration-free** — the last-seen `sleepEnd` + idempotency live in the existing `analyses` table (`analysis_type='wake_check'`, one row per (user, day), updated in place; `context_packet.sleepEndUtc` carries the value), and the verdict's once-per-day guarantee reuses the existing morning-analysis idempotency (`generate_and_store` short-circuits per `subject_date`). **Wiring:** replaces the 06:30 `morning_weather_sync` cron in `create_scheduler()` with the `wake_check` interval job (15 min, seeded `next_run_time = now + 3 min`) + the `morning_backstop` cron; adds `wake-check` to `run_scheduled.JOBS` so an external cron can drive it too. *Why now:* settled by DECISIONS #86 — App Sleeping is off so the container is always-on and in-process APScheduler fires reliably and handles BST↔GMT via `timezone=Europe/London` (a UTC external cron would drift ±1 h on the window/backstop bounds, so wake detection stays in-process; the `run_scheduled` entry is resilience, not the primary path). *Why reuse `run_morning_weather_sync` unchanged:* only the **trigger** changes, not the verdict — the sync/analysis/Amber-regen pipeline and its per-profile idempotency are untouched, so this is a thin integrator over proven code. **Known single-subject simplification:** `run_morning_weather_sync` serves *all* active profiles, so when any one profile is ready the whole sync fires; for a 2nd user who has not yet woken that means a verdict on whatever data exists today (and the morning-idempotency then blocks a later fresh one). Acceptable for this 1–2 user app (effectively just Mark); a per-profile trigger would require changing `run_morning_weather_sync`, which the design deliberately keeps unchanged. **Calibration constants** (`WINDOW_START/END`, `BACKSTOP`, `DURATION_FLOOR_MIN`, `SETTLE_MIN`) are tuned to Mark's distribution and live in `wake_detection.py`; retune there if his schedule shifts. Full design + decision matrix in `docs/designs/wake-triggered-morning.md`.

A non-bike session shows the same card with Edit/Swap/Skip but **no Zwift upload**. **Delivery decision (supersedes #31 lead-days=2 *and* #91 same-day-on-approval):** push each workout to Zwift as soon as it is in the plan. Because events then live in Zwift for days, the actions map to event mutations — plan-set/restructure → **create**; Edit / Approve → **replace** (with the manual or adjusted IR); Swap day → **move** both affected events; Skip → **delete**; Ignore → **no-op** (the original is already there, so "ignore" is a pure client-side dismiss). The intervals.icu rail today is only create+delete (Batch 12); Batch 29 adds **replace** + **move**, keyed idempotently to `planned_workout_id`+`version` so re-syncs never duplicate. **Four forks settled this session:** (a) **Swap = unified move-or-swap** — pick a target day; empty → move, occupied → swap (Mark was happy with either, one control does both); (b) **Ignore = pure dismiss**, no backend call; (c) **Skip = mark-only** — a `planned → skipped` transition on the already-free `planned_workouts.status String(50)` (**no migration**) plus the Zwift delete, no reshuffle "for now" (reshuffling stays the separate `/restructure` tool); (d) **Universal card** — change `useDailyPhase` so any planned workout (not just bike) leads the card, `rest_day` only when nothing is planned. **Policy reversal recorded here:** push-on-plan-set delivers the *as-planned* baseline **without per-workout human approval**, deliberately reversing #29/#30's "nothing delivered silently" for the baseline — approval now gates **only the morning adjustment**. **Invariants preserved:** Red-never-VO2 (`blocks_red_vo2`, #30) still gates Approve; every mutation is audited as an `analyses` row (Batch 13 pattern); a failed Zwift re-sync must leave local state honest (#97 drive-then-disable). **Implementation status:** built on `feat/batch-29-today-card-actions` on 2026-06-29 using true intervals.icu update-in-place for replace/move, exposed through the Today-card action routes and the daily-loop delivery state.
