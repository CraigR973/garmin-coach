### 2026-06-21 — Phase 2 Batch 14 (dynamic weekly restructuring)
63. **The weekly restructurer is a deterministic permutation engine over the week's bike sessions, with the no-stack rule as a hard constraint and defer-on-fatigue as a lexicographic objective.** `plan_week_restructure` (in `services/weekly_restructure.py`) reassigns the week's bike workouts among the dates that already hold a bike session (strength/mobility days stay put and act as spacers), enumerates the permutations (capped at `MAX_PERMUTE_SESSIONS`), and picks the lowest-cost feasible arrangement. **Feasibility** = no VO2↔Sweet-Spot pair within `MIN_GAP_DAYS` (same/adjacent days). When **fatigued**, the cost key is `(infeasible?, defer_cost, moves, shift)` so deferring hard sessions later in the week is the primary objective; when fresh it is `(infeasible?, moves, shift, defer_cost)` so a conflict is fixed with minimal disruption and an already-spaced fresh week is a no-op. *Why:* the spacing/fatigue rules are documented coaching constraints (KB `training_plan.constraints`, ARCHITECTURE §6) and a small enumerated search makes them inspectable, deterministic, and unit-testable without a DB or an LLM — the engine is a pure function and the DB service is a thin wrapper. The fatigue signal itself (`assess_recovery_signal`) reads the latest readiness, HRV status, and the morning-verdict trend (a Red, ≥2 Ambers in the window, low readiness, or unbalanced HRV).

Following #71 the engine is **human/API-triggered**: `GET /api/v1/trends/seasonal`, `GET /api/v1/trends/year-on-year`, and `GET /api/v1/trends/narrative` preview and **never write**; only `POST /api/v1/trends/narrative/run` records. The narrative is **idempotent per window** (`subject_date` = canonical window start) and is disambiguated month-vs-season by a per-bucket `prompt_version` (`trends-month-v1` / `trends-season-v1`), since both can share a start date (e.g. summer and June both start 2026-06-01). **When history is insufficient the run reports `insufficient_history` deterministically and never calls the model.** A read-only Trends PWA tab (`/trends`) renders the YoY deltas, recent windows, and the on-demand narrative. The Batch 20 `AnthropicReviewClient` gained an optional `system_prompt` override (backward-compatible) so Batches 21/23 reuse the boundary with their own prompt. *Why:* the seasonal lens is inherently *narrative* like Batch 20, so it reuses the inspectable, fakeable-in-tests boundary for prose while every number stays deterministic and unit-testable (the v3 boundary rule); honouring #45 in the aggregation and surfacing sample/excluded counts keeps comparisons trustworthy; graceful insufficient-history (never inventing a prior-year number) matches #71's sample-size gates; and reusing `analyses` keeps the batch migration-free (mirrors #62/#71/#80/#81).

### 2026-06-23 — Phase 3 Batch 22 (hypothesis evaluation)

83. **Hypothesis evaluation is a *deterministic, advisory* engine that reuses the Batch 17 insights math (`_slope`/`pearson`/`compute_drivers`) to recommend a conclusion per experiment, but never changes status — concluding stays the human-gated, terminal `POST /…/status` action (#72). No LLM, no migration; reuses `experiments`/`analyses`.** `services/experiment_evaluation.py` dispatches on the experiment's `slug` (set by the Batch 17 tracker) to one of three pure, DB-free evaluators, each surfacing its evidence window and reasons: **gate** (`collagen`) — counts the current run of *calendar-consecutive* nights at/above the age-adjusted sleep floor (`Sleep.age_adjusted_score`, falling back to `score`); a met gate (≥`gateNights`) recommends `supported`, else `inconclusive`. **correlation** (`early_waking_0400`) — Pearson-ranks the *measured* candidate drivers (overnight low °C from `weather_daily`, sleep stress) against an overnight-disruption proxy (`awake_sleep_sec`→minutes) via `compute_drivers`; |r|≥0.5 → `supported` (identifiable trigger), ≥0.3 → `inconclusive`, otherwise `refuted` (no measured trigger); drivers the hypothesis names but we don't capture (alcohol, late snack) are surfaced as a coverage gap, not silently dropped. **group_compare** (`recovery_week_disruption`) — before/after comparison of mean age-adjusted sleep on recovery-week vs build-week nights, labelled from the containing `plan_blocks.block_type`; recovery worse by ≥3 points → `supported`, better → `refuted`, inside the band → `inconclusive`. Every evaluator **skips with an explicit reason** below its sample gate (#71): gate needs ≥5 scored nights, correlation ≥8 paired nights, group compare ≥4 per group. User-created experiments fall back to the correlation evaluator when they declare `candidateDrivers`, else report `no_evaluator` ("conclude manually"). The recommendation maps onto the tracker's outcome vocabulary (`supported`/`refuted`/`inconclusive`) so **accepting it flows through the existing conclude path** unchanged. Following #71 the surface is human/API-triggered: `GET /api/v1/experiments/{id}/evaluate` previews and **never writes**; `POST /api/v1/experiments/{id}/evaluate/run` records an audit row in `analyses` (`analysis_type='experiment_evaluation'`, `prompt_version='experiment-eval:v1'`), **idempotent per (experiment, subject date)** (a row already present for that date is reused unless `force=True`; idempotency is keyed on `context_packet.experimentId` since many experiments share a date). The router exposes `canConclude` (true only on a non-concluded experiment with an actionable recommendation) so the PWA only offers the terminal action when it's legal. A new read-only Experiments PWA tab (`/experiments`, "Tests") lists the hypotheses and, per card, evaluates the evidence and offers "Conclude as <recommendation>" wired to `POST /…/status`. *Why:* unlike the narrative batches (20/21), evaluation is a *safety/accuracy* judgement, so — like Batches 13/14/16/17/19 — it's inspectable, unit-testable arithmetic that works without `ANTHROPIC_API_KEY`; reusing the Batch 17 slope/Pearson math avoids a second copy of the insight stats; the never-auto-conclude guard (no status write anywhere in the engine or `/evaluate`) is explicitly unit-tested, keeping `concluded` human-gated and terminal (#72); and reusing `experiments`/`analyses` keeps the batch migration-free (mirrors #62/#71/#72/#80/#81/#82). The standing hypotheses' alcohol/late-snack triggers aren't captured by any synced source, so the correlation evaluator reports them as an unmeasured-coverage gap rather than pretending to rule them out.
