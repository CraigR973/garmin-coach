## v2 batch plan

Batch ledger for Garmin Coach v2 — "the coach". Decomposed from `ARCHITECTURE.md`
§6 (v2 roadmap) and `DECISIONS.md` #25–33. Same conventions as v1: a batch is
shipped only after `/closeout` finishes; leave the row unstruck and `Planned`
until started.

**Sequencing rationale.** The **Zwift delivery rail (Batch 12)** is the
foundational dependency for everything else in v2 — executable coaching, dynamic
restructuring, holiday resume, and app-generated blocks all assume the app can
push a structured workout — so it lands first among the substantive batches.
Each later batch delivers its output *through* that rail under the
propose → approve → push contract (Decision #29).

**Phase 1 debt decision.** The Phase 1 retrospective items are scheduled as a
**standalone clean-up Batch 11 first**, ahead of the substantive v2 work. The
driver is the WC2026 "player" → user rename (Decision/debt #3): it touches
`auth.py`, `models/profile.py`, `models/notification.py`, and router error
details, and grows more expensive with every endpoint v2 adds — so it is cheapest
to clear before Batch 12 onward expand the API surface. The smaller items
(ForgotPin flow, dead `email.py`, `score-input.tsx` JSDoc, residual "predictions"
offline-queue refs) ride along in the same batch.

**Production daily-loop gate.** The 2026-06-21 production smoke check proved the
API/auth/daily-loop surface is live for Mark, but the actual daily data loop is
not yet healthy: Mark had to be seeded, the response had today's
`subjectDate=2026-06-21` but `morningAnalysis`, Garmin daily metrics/sleep,
Hive thermal values, weather, and analyses were all empty, and the first
post-seed Hive poll logged `hive temperature poll failed`. Railway production
was also missing the expected data/analysis environment variables
(`GARMIN_EMAIL`, `GARMIN_PASSWORD`, `GARMIN_TOKENSTORE`, `HIVE_EMAIL` /
`HIVE_PASSWORD`, `ANTHROPIC_API_KEY`) and logged `environment="development"`
because `ENVIRONMENT=production` was not set. Treat this as the first Batch 12
carry-in before adding new v2 delivery surfaces: production must show real
Garmin + Hive + weather + morning-analysis data for Mark, not just HTTP 200 and
login success.

**Update (2026-06-21 closeout).** Batch 12 shipped the delivery rail, the Garmin
`GARMIN_TOKENSTORE_B64` token-blob auth, and the Anthropic fail-closed validator,
but the strict data gate is still **not** met: production has no Garmin
daily-metrics/sleep sync — `GarminSyncService.sync_daily` is defined (Batch 2)
but has no caller in `src/`, so today's daily metrics + sleep are never synced
and the morning verdict has no readiness/sleep input. The data gate is relocated
to **Batch 18** (DECISIONS #57).

## v3 batch plan

Batch ledger for Garmin Coach v3 — "the long game". Decomposed from
`ARCHITECTURE.md` §6 (v3 roadmap). Same conventions as v1/v2: a batch is shipped
only after `/closeout` finishes; leave the row unstruck and `Planned` until
started. One batch per v3 roadmap bullet, capstone last. Decision numbers are
assigned at `/batch-start` (next free is #80).

**Narrative reviews use the Claude boundary; rollups stay deterministic.** Unlike
the deterministic engines of Batches 14/16/17 (#63/#69/#71), v3's reviews, the
trends narrative, and the handover export are inherently *narrative*, so they
reuse the thin Anthropic Messages boundary with stored prompt/version metadata in
`analyses` (the Batch 6/8 pattern, #47) — fakeable in tests without
`ANTHROPIC_API_KEY`. The *rollup packets* that feed them stay deterministic and
unit-testable.

**Human/API-triggered, not new scheduler crons.** Following #64/#71, reviews,
trends, and the handover export generate on demand via `POST /…/run` and record to
`analyses`; `GET` previews never write. An optional scheduled `/run` can be layered
on later without changing the engines — no silent background rewriting and no
racing the 06:30 pipeline.

**Migration-free where possible.** Outputs land in `analyses` under new
`analysis_type` values (`weekly_review`, `monthly_review`, `seasonal_trend`,
`experiment_evaluation`, `handover_export`); structured state reuses
`knowledge_base` / `metric_baselines` (#44/#68/#70/#71/#72). Any batch that
genuinely needs a new table must justify it.

**Sequencing.** Batch 19 (strength watching-brief) is the self-contained, no-LLM
opener and preserves the #49 recovery-isolation invariant. Batch 20 establishes
the v3 review boundary that Batches 21 and 23 reuse. Batch 22 *extends* the Batch
17 experiment tracker (#72) rather than replacing it. Batch 23 (handover export) is
the integrative capstone (#13) and lands last because it summarizes the output of
every batch before it.

**Seasonal / year-on-year degrades gracefully.** True year-on-year needs ≥1 year
of history; the 84-night backfill seeds from ~24 Mar 2026, so real YoY comparisons
mature ~Mar 2027. Batch 21's engine ships now and reports insufficient history
until the window fills, consistent with #71's sample-size gates.

## Post-roadmap — passive-first loop batch plan

Batch ledger for the **passive-first reassessment** (2026-07-03) — re-reading the
app against Mark's refined user story: two nested feedback loops (**last night's
sleep → today's training**, daily; and a **completed multi-week block → the next
block**, per block), run on the principle **"effortless, not empty"** — the app
does the work and reaches out, while the rich data stays there for Mark to browse
(he enjoys it). Same conventions as v1–v3: a batch is shipped only after
`/closeout` finishes; leave the row unstruck and `Planned` until started. Decision
numbers assigned at `/batch-start` (next free is **#117**).

**What already exists (the reassessment's finding).** Most of the loop is built:
the wake-triggered morning verdict (sleep → readiness → sleep-adjusted plan via
`executable_coaching`), per-session post-workout reads for all five modalities
(ride / strength / flexibility / walk + the breathwork brief), and — crucially —
the real-world actuation Mark wants: the **Batch 27 Dreo fan autopilot already
drives the bedroom temperature overnight**. The corrected "effortless, not empty"
principle is also already well served by the Home (collapse-not-remove) + Trends /
Reviews / bedroom chart / interval-execution split. The structure is sound; these
batches are **connective tweaks, not a rebuild.**

**The gaps these batches close.** (1) The two highest-value outputs — the morning
verdict and each post-workout analysis — are generated silently and **never
pushed**; only the static 20:00 nudge + thermal/stale alerts push, so the app is
browse-first when the story wants push-first. (2) The evening thermal nudges still
tell Mark to hand-cool the room as if the fan autopilot doesn't exist. (3) There
is **no projection of how today's training will affect tonight's sleep** (the
"Tonight" section is the static protocol). (4) **Block-to-block progression
doesn't exist** — `generate_block_plan` is profile/FTP-only and never reads how
the last block went. (5) The loop is **implicit** — `DailyPhase` is `pre_ride |
post_ride | rest_day` (ride-shaped, no evening phase), which is why the pushes are
piecemeal.

**Sequencing.** **Batch 45** (proactive push + fan-reconciled nudges) ships first
— smallest, highest-impact, turns the app push-first and fixes the fan/nudge
contradiction. **Batch 46** (evening sleep projection) and **Batch 47**
(block-to-block progression) build the two missing loop stages and can proceed in
parallel — 46 closes the daily loop back to sleep, 47 builds the second, per-block
coupling. **Batch 48** (explicit loop model) is the **optional consolidating
refactor** that makes the loop first-class — deferrable, sequenced last, worth
doing only once the piecemeal wiring of 45–47 costs more than the refactor. Tiers
per #19: all 🔴 High (scheduler edge cases, analysis-engine reasoning, verdict /
plan logic, a shipped-model revision).

**Batch 45 (Proactive push + fan-reconciled nudges)** turns the app push-first.
The morning verdict and each post-workout analysis are generated by the scheduler
but **never pushed** — only the static 20:00 nudge + thermal/stale alerts reach
Mark — so today he must open the app to find his verdict/read. This wires a push
into the two existing completion points (`run_morning_weather_sync` after
`generate_and_store`; the hourly poll after `generate_for_pending_*`), reusing
`send_notification` + the `_send_once` `analyses`-tag idempotency so each
verdict/analysis pushes **exactly once** (09:30-backstop + regeneration safe). It
also reconciles the evening thermal nudges with the Batch 27 fan autopilot: defer
the manual pre-cool/seal nudges when `fan_auto_enabled` and the loop is
applying/holding, and escalate a "check the fan" push **only** when the fan can't
cope (`unreachable` / `no_data` / can't-reach-target) — the "notify only if he must
physically act" model. No migration. 🔴 High. Full spec:
`docs/designs/proactive-push-fan-reconciled.md`.

**Batch 46 (Evening sleep projection)** closes the daily loop **back to sleep**.
Today the forward-looking read is training-only ("impact on tomorrow") and the
"Tonight" Home section is the static protocol; nothing projects how today's
training will affect tonight's sleep. A deterministic, advisory `sleep_projection`
derives a qualitative directional read (**no numeric score** — false precision, per
#114) from today's training load + Mark's own measured sleep drivers (Batch 17/34)
+ the overnight weather, and layers 1–2 personalised prep actions onto the static
KB protocol, surfaced in the Tonight section (and optionally pushed via Batch 45).
Advisory only — never touches the verdict or the fan thresholds (#71/#72); degrades
to the plain protocol on a rest day / before driver history exists. 🔴 High. Full
spec: `docs/designs/evening-sleep-projection.md`.

**Batch 47 (Block-to-block progression)** builds the second feedback coupling Mark
named — **completed multi-week block → next block**. `generate_block_plan` is
profile/FTP-only and never reads how the last block went, though the signals are
all shipped (Batch 17 FTP-drift, Batch 20 rollups, Batch 44 execution grades,
`manual_entries` adherence). A deterministic `block_progression` core aggregates
the completed block's load / adherence / execution / drift / trend into a
next-block proposal (recommended FTP + focus carry-forward) that **seeds** the
generator draft with its evidence — the athlete still refines-then-locks
(#16/#69), nothing auto-applies. Reuses the existing engines, no new stats, no
LLM, no migration. 🔴 High. Full spec:
`docs/designs/block-to-block-progression.md`.

**Batch 48 (Explicit daily/block loop model) — optional, consolidating.** The loop
is implicit across scheduler jobs, Home sections, and services, which is why the
pushes are hand-wired and why `DailyPhase` is ride-shaped (`pre_ride | post_ride |
rest_day`, `post_ride` off ride analyses only, no evening phase). This
behaviour-preserving refactor makes the day/block loop a first-class state model
with an orchestration seam ("where is Mark + what's next") that Batches 45–47 hook
into, generalises the phase to any modality (a strength/walk-only day advances to
post-training), and gives evening a real `wind_down` phase instead of a 20:00 clock
nudge. **Deferrable** — 45–47 ship without it; sequenced last, when the piecemeal
wiring costs more than the refactor. No new coaching logic, no migration. 🔴 High.
Full spec: `docs/designs/explicit-loop-model.md`.
